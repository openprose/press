# v1.3.0 Changes Analysis -- Run 017

## Summary

Run 017 scored 0 on a single-task arc-compound session (task 0934a4d8). Only 2 root iterations were used out of a budget of 10. The root cause is that **Change 1 (Tool-Call REPL Driver) was not activated** -- the eval harness routed through OpenRouter using the text-block code path, so both agents produced massive multi-block responses. All code blocks executed in a single iteration, collapsing the multi-iteration exploration cycle into one turn and triggering cascading failures.

Change 2 (IoC Program Loading) worked correctly at the mechanical level.

---

## Change 1: Tool-Call REPL Driver

### What was supposed to happen

The Anthropic Messages API driver (`src/drivers/anthropic-messages.ts`) uses a single `execute_code` tool with `disable_parallel_tool_use: true` to guarantee exactly one code block per LLM response. This should eliminate multi-block hallucination.

### What actually happened: THE DRIVER WAS NOT USED

**Evidence:** `eval/run.ts` line 263-291 -- the `resolveCallLLM()` function always returns `fromOpenRouter()`. It never calls `fromAnthropicMessages()`, regardless of model spec. The eval harness has no code path that selects the Anthropic Messages driver.

```typescript
// eval/run.ts:263-291
function resolveCallLLM(spec: string, ...): { callLLM: CallLLM; displayName: string } {
    // ...
    if (parts[0] === "openrouter") {
        return { callLLM: fromOpenRouter(...), ... };
    }
    return { callLLM: fromOpenRouter(spec, apiKey, overrides), ... };
}
```

The model spec `anthropic/claude-opus-4-6` went through `fromOpenRouter()`, which uses the text-block code path (markdown ` ```javascript ` fences in plain text). The tool-call driver exists in `src/drivers/anthropic-messages.ts` but is never imported or called from the eval harness.

### Consequence: Catastrophic multi-block execution

Without the tool-call enforcement, Claude Opus 4.6 generated massive multi-block responses at every level:

**Root agent, iteration 0:** 5 code blocks in one response:
1. Session setup (prints task info, initializes `__arcSession`)
2. Full task processing (delegates to solver, reads result, sanity-checks, submits, curates)
3. Pass@2 transition check
4. Pass@2 retry logic
5. Final return

All 5 blocks executed sequentially in a single iteration. The model planned the entire session lifecycle in advance, hallucinating intermediate outputs between blocks. This consumed the entire pass@1 AND pass@2 logic in one turn.

**Pass@1 solver child (D1), iteration 0:** 6 code blocks in one response:
1. Data exploration (print grids)
2. 3x3 block analysis
3. Block overlay analysis
4. Block property analysis (all-zero vs mixed)
5. Block color/nonzero analysis
6. Minority color hypothesis (truncated -- hit output token limit)

All 6 blocks ran in one "iteration". The solver consumed 6 steps of exploration blind, since blocks 2-6 were written before seeing the output of block 1.

**Pass@1 solver child (D1), iteration 1:** 4 code blocks:
1. Find 8-region, check dimensions
2. Check symmetries, try 180-rot fill
3. Verify transform on all training pairs + LOO
4. Write taskLog + return

The solver hallucinated "Perfect! 180-degree rotational symmetry works for ALL training pairs" as free text between blocks 2 and 3. It then wrote verification code in block 3 and the return in block 4. When block 3 actually ran, it printed `Training pair 0: WRONG` through `Training pair 3: WRONG` and `All correct: false`. But block 4 (the return) was already written and still executed, calling `return(JSON.stringify({ solved: true, confidence: 1.0, answer: testOutput }))`.

**This is the canonical multi-block hallucination failure.** The solver claimed `solved: true` with `confidence: 1.0` despite ALL training pairs failing verification. The orchestrator trusted this, submitted a wrong answer, and burned submission 1.

**Pass@2 retry solver (D2):** Same pattern. Multiple blocks per iteration, hallucinated verification results. Found horizontal mirror symmetry (which is also wrong for this task), returned `solved: true` again, burned submission 2. Score: 0.

### Evidence of multi-block harm

1. **Root trace, iteration 0, `code` array:** Contains 5 elements (lines 21-26 of the result JSON). The engine ran all 5. The output contains the full session: setup, delegation, pass@1 submission, pass@2 transition, retry submission, and final return -- all in one output string.

2. **D1 solver trace, iteration 1, `code` array:** Contains 4 elements (lines 60-63). Block 4 calls `return()` with `solved: true`. The solver's own output shows `All correct: false` and `LOO FAIL: held out pair 0`. The return was written before seeing these outputs.

3. **D2 solver trace, iteration 0, `code` array:** Contains 5 elements (lines 88-92). Follows the same pattern as D1.

4. **Unawaited rlm() warning:** The output includes `[ERROR] 1 rlm() call(s) were NOT awaited`. This happened because multiple code blocks executed: one block wrote the `rlm()` call, and by the time the next block ran, the promise was unresolved. In a tool-call world with one block per response, the `await` would be properly handled within the single block.

### What the tool-call driver would have prevented

- Each response would contain exactly 1 tool use (1 code block)
- The model would see real execution output before writing the next block
- Verification failure (`Training pair 0: WRONG`) would be visible before the return block
- The solver could not claim `solved: true` after seeing `All correct: false`
- The orchestrator would iterate through tasks one at a time instead of scripting the entire session in advance

### Assessment

**Change 1 was not tested.** The code exists and appears correct (`src/drivers/anthropic-messages.ts` implements the tool-call protocol properly), but it was never wired into the eval harness. The `resolveCallLLM()` function needs an Anthropic-specific branch that calls `fromAnthropicMessages()` when the model spec matches `anthropic/*`.

## Change 2: IoC Program Loading

### What was supposed to happen

The `--program arc2-compound` flag loads:
- `root.md` body as `globalDocs` (visible at all depths)
- `orchestrator.md` as `rootAppBody` (root agent's `<rlm-program>`)
- `solver.md` registered in `childApps` under `arc2-solver` and `solver`

### What actually happened: Loading worked correctly

**Evidence from the trace:**

1. **Root agent received orchestrator program.** The root's first code block follows the orchestrator.md template exactly: initializes `__arcSession`, reads `__arcTaskIds`, prints library state. The delegation calls `rlm(query, undefined, { app: "arc2-solver", model: "intelligent", maxIterations: 18 })` which matches the program's documented pattern.

2. **`{ app: "arc2-solver" }` resolved correctly.** The solver children ran and had access to the correct task data via `globalThis.__arcCurrentTask` and `globalThis.__arcTasks`. The solver's behavior (printing training pairs, analyzing grids, writing transforms, pushing to `__arcLibrary.taskLog`) matches the solver.md program precisely. `loadProgram()` registers under both `arc2-solver` (the frontmatter `name`) and `solver` (the filename). The orchestrator used `arc2-solver`, which resolved.

3. **globalDocs were visible to the solver.** The solver child's trace shows it accessed `globalThis.__arcLibrary` (the shared library from the `&Library` state schema defined in root.md). The component catalog, shared state schemas, and composition principles from root.md were available as globalDocs. The solver checked library primitives and strategies as instructed: `console.log('Library: ${Object.keys(library.primitives).length} primitives, ${library.strategies.length} strategies')`.

4. **Shared sandbox state worked correctly.** The solver wrote to `__arcLibrary.taskLog`, and the orchestrator read it back: `const logEntry = library.taskLog.filter(e => e.id === taskId).pop()`. The `&Library` pass-by-reference pattern functioned as designed.

### What did not go well

1. **Solver reported output as `sameSize: true` on the retry.** The task has 30x30 inputs and varying output sizes (9x4, 4x5, 3x7, 4x4). The retry solver (D2) incorrectly set `sameSize: true` in its `structuralProps` and returned a 30x30 grid instead of the 9x3 region. This is not a program loading issue -- it is a consequence of multi-block execution preventing the solver from seeing its own verification output.

2. **globalDocs may have been too verbose for the solver.** The root.md body contains 284 lines of component catalog, composition principles, and state schemas. The solver only needs the `&Library` schema and harness globals. The composition principles (CURATION IS THE RETURN ON COMPOSITION, COLLAPSE IS THE DEFAULT FAILURE MODE, etc.) are relevant to the orchestrator, not the leaf solver. This adds unnecessary context to the solver's system prompt. However, there is no evidence this caused confusion in this run -- the solver followed its own program correctly.

3. **The `model: "intelligent"` parameter in the orchestrator's delegation.** The orchestrator program says to delegate via `rlm(goal, null, { app: "arc2-solver" })` but the actual call was `rlm(query, undefined, { app: "arc2-solver", model: "intelligent", maxIterations: 18 })`. The `model: "intelligent"` parameter selects a model alias, but since the model aliases are built from the harness (not the program), this depends on the eval harness configuration. This worked in this run because the model aliases were configured, but it is an implicit dependency not declared in the program.

### Assessment

**Change 2 worked as designed.** The program loading pipeline (`loadProgram()` -> `globalDocs` + `rootAppBody` + `childApps`) functioned correctly. The orchestrator received its program, the solver received its program, `app: "arc2-solver"` resolved, and shared sandbox state (`&Library`) worked. The run's failure is entirely attributable to Change 1 not being active.

   -> Submission 2 wasted, 0 submissions remaining

## Recommendations

### 1. Wire the Anthropic Messages driver into the eval harness (CRITICAL)

The `resolveCallLLM()` function in `eval/run.ts` must detect `anthropic/*` model specs and use `fromAnthropicMessages()` instead of `fromOpenRouter()`. Without this, Change 1 has zero effect.

Suggested implementation:
```typescript
import { fromAnthropicMessages } from "../src/drivers/anthropic-messages.js";

function resolveCallLLM(spec: string, ...): { ... } {
    const parts = spec.split("/");
    const provider = parts[0];

    if (provider === "anthropic") {
        const apiKey = process.env.ANTHROPIC_API_KEY;
        if (!apiKey) { /* error */ }
        const modelId = parts.slice(1).join("/");
        return {
            callLLM: fromAnthropicMessages({
                baseUrl: "https://api.anthropic.com/v1",
                apiKey,
                model: modelId,
                ...modelOverrides(spec),
            }),
            displayName: `${modelId} (anthropic-messages)`,
        };
    }
    // ... existing OpenRouter path
}
```

### 2. Add a `maxBlocksPerIteration: 1` fallback for OpenRouter

Even if the Anthropic direct driver is not used, the existing `maxBlocksPerIteration` option can enforce single-block execution on the text-block path. The harness should default to `maxBlocksPerIteration: 1` when no tool-call driver is active. This is a safety net, not a replacement for the tool-call driver, since it still wastes the model's output tokens on discarded blocks.

### 3. Consider splitting globalDocs by audience

The root.md body contains content relevant to the orchestrator (composition principles, component catalog) and content relevant to all agents (state schemas, harness globals). The solver does not need COLLAPSE IS THE DEFAULT FAILURE MODE or BUDGET PROPORTIONALITY. Consider splitting root.md so that only shared schemas go into globalDocs, while orchestrator-specific principles stay in orchestrator.md.

### 4. Do not roll back either change

- **Change 1** is the right design. It just was not connected. Once wired in, it mechanically eliminates the exact failure mode observed in this run.
- **Change 2** worked correctly and provides clean separation between program structure (root.md), orchestrator behavior (orchestrator.md), and solver behavior (solver.md). The IoC pattern is strictly better than the old `--app` + `--child-app` approach.
