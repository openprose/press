# Interleaved Thinking Implementation Plan

## Problem Statement

After migrating from text-block extraction (models wrote fenced `\`\`\`javascript` blocks) to proper tool calls (`execute_code` tool), **all reasoning fields in traces are empty strings**. The causal chain:

1. `TOOL_CHOICE` forces the `execute_code` function (named tool choice).
2. In the OpenAI Chat Completions format, when `tool_calls` are present the provider returns `content: null`.
3. The driver reads `choice.message.content ?? ""` -- always `""`.
4. That empty string flows into `CallLLMResponse.reasoning`, then `TraceEntry.reasoning`, then the viewer shows nothing.

The model IS reasoning internally before emitting the tool call, but that reasoning is discarded by the response format. OpenRouter's **Reasoning Tokens** feature makes this thinking visible as a separate field in the response.

## How OpenRouter Reasoning Tokens Work

### Request Side

Add a `reasoning` parameter to the request body:

```json
{
  "model": "anthropic/claude-opus-4-6",
  "messages": [...],
  "tools": [...],
  "tool_choice": { "type": "function", "function": { "name": "execute_code" } },
  "reasoning": {
    "effort": "high"
  }
}
```

The `reasoning` parameter is a `ChatGenerationParamsReasoning` object with:

| Field     | Type                                                     | Purpose                                                                |
|-----------|----------------------------------------------------------|------------------------------------------------------------------------|
| `effort`  | `"xhigh"` / `"high"` / `"medium"` / `"low"` / `"minimal"` / `"none"` (nullable) | Controls reasoning depth. Maps to provider-native budget allocation. |
| `summary` | `"auto"` / `"concise"` / `"detailed"` (nullable)        | Controls reasoning output verbosity.                                   |

For Anthropic models specifically, `effort` maps to `budget_tokens` allocation:
- `high` = 80% of `max_tokens`
- `medium` = 50%
- etc.
- Minimum 1024 tokens, max 128,000 tokens.

There is also an older `include_reasoning: true` parameter that still works but the `reasoning` object is the modern unified interface.

### Response Side

The response message gains two new fields alongside `content` and `tool_calls`:

```json
{
  "choices": [{
    "message": {
      "role": "assistant",
      "content": null,
      "reasoning": "Let me analyze the grid...\nI see a pattern...",
      "reasoning_details": [
        {
          "type": "reasoning.text",
          "id": "rd_abc123",
          "format": "anthropic-claude-v1",
          "index": 0,
          "text": "Let me analyze the grid...\nI see a pattern..."
        }
      ],
      "tool_calls": [{
        "id": "call_abc123",
        "type": "function",
        "function": {
          "name": "execute_code",
          "arguments": "{\"code\":\"console.log('hello')\"}"
        }
      }]
    },
    "finish_reason": "tool_calls"
  }]
}
```

Key fields:
- **`message.reasoning`** (string | null): Plain-text reasoning. This is what we need for `CallLLMResponse.reasoning`.
- **`message.reasoning_details`** (array | null): Structured reasoning blocks with metadata, type info, and signatures. Needed for round-tripping.

### Round-Trip: Sending Reasoning Back

When constructing conversation history for the next turn, reasoning must be sent back on the assistant message so the model can continue from its previous reasoning state:

```json
{
  "role": "assistant",
  "content": null,
  "reasoning": "Let me analyze the grid...",
  "tool_calls": [{ ... }]
}
```

Or with full structured details:

```json
{
  "role": "assistant",
  "content": null,
  "reasoning_details": [{ ... original blocks ... }],
  "tool_calls": [{ ... }]
}
```

**Critical constraint**: "The entire sequence of consecutive reasoning blocks must match the outputs generated by the model during the original request." You cannot modify, reorder, or truncate them.

## Current Architecture (What Exists)

### Data Flow

```
OpenRouter API Response
    -> choice.message.content ?? ""           // ALWAYS "" when tool_calls present
    -> CallLLMResponse { reasoning, code, toolUseId }
    -> TraceEntry { reasoning, code[], output, error }
    -> viewer.html renders entry.reasoning
```

### Message History Round-Trip

The engine (rlm.ts) stores messages as `Array<{ role: string; content: string }>` -- flat strings only. Tool-call metadata is serialized into these strings using markers:

```
Assistant: "__TOOL_CALL__\n{toolUseId}\n{reasoning}\n__CODE__\n{code}"
User:      "__TOOL_RESULT__\n{toolUseId}\n{output}"
```

The driver's `translateMessages()` function reconstructs proper `ChatMessage` objects from these markers before sending to the API. This is the key indirection point -- reasoning needs to survive this round-trip.

## Implementation Plan

### Step 1: Add `reasoning` to Request Body

**File**: `src/drivers/openrouter-compatible.ts`

In the `fromOpenRouterCompatible` function, add the `reasoning` parameter to the request body.

```typescript
const reqBody: Record<string, unknown> = {
    model,
    messages: chatMessages,
    max_tokens: maxTokens,
    tools: [EXECUTE_CODE_TOOL],
    tool_choice: TOOL_CHOICE,
    parallel_tool_calls: false,
    reasoning: {                    // NEW
        effort: "high",
    },
};
```

**Design decision -- effort level**: Start with `"high"` (80% of max_tokens). This gives substantial thinking budget without consuming the entire token allocation. Consider making this configurable via `OpenRouterCompatibleOptions` later, but `"high"` is the right default for agentic workloads.

**Design decision -- should effort be configurable?** Yes, add an optional `reasoningEffort` field to `OpenRouterCompatibleOptions`. Default to `"high"`. This lets the eval harness or CLI tune it per-model.

```typescript
export interface OpenRouterCompatibleOptions {
    // ... existing fields ...
    /** Reasoning effort level (default "high"). Set to "none" to disable. */
    reasoningEffort?: "xhigh" | "high" | "medium" | "low" | "minimal" | "none";
}
```

### Step 2: Parse `reasoning` from Response

**File**: `src/drivers/openrouter-compatible.ts`

Update the `ChatCompletionResponse` interface to include reasoning fields:

```typescript
interface ChatCompletionResponse {
    choices: Array<{
        message: {
            content: string | null;
            reasoning?: string | null;           // NEW
            reasoning_details?: Array<{          // NEW
                type: string;
                id?: string;
                format?: string;
                index?: number;
                text?: string;
                summary?: string;
                data?: string;
                signature?: string;
            }> | null;
            tool_calls?: Array<{
                id: string;
                type: "function";
                function: { name: string; arguments: string };
            }>;
        };
        finish_reason?: string;
        native_finish_reason?: string;
    }>;
    error?: { message: string; code?: number };
}
```

Update the response parsing (currently line 229):

```typescript
// BEFORE:
const reasoning = choice.message.content ?? "";

// AFTER:
const reasoning = choice.message.reasoning
    ?? choice.message.content
    ?? "";
```

This cascading fallback ensures:
1. If `reasoning` field is present (OpenRouter with reasoning enabled) -- use it.
2. If only `content` is present (non-OpenRouter providers, or reasoning disabled) -- use it.
3. If both are null (tool_calls present, no reasoning) -- empty string.

Also capture `reasoning_details` for round-tripping:

```typescript
const reasoningDetails = choice.message.reasoning_details ?? null;
```

### Step 3: Extend `CallLLMResponse` to Carry `reasoningDetails`

**File**: `src/rlm.ts`

```typescript
export interface CallLLMResponse {
    reasoning: string;
    code: string | null;
    toolUseId?: string;
    /** Structured reasoning blocks for round-tripping to the API. */
    reasoningDetails?: Array<Record<string, unknown>> | null;  // NEW
}
```

Using `Array<Record<string, unknown>>` keeps `rlm.ts` decoupled from OpenRouter-specific types. The driver produces them; the engine stores and round-trips them opaquely.

### Step 4: Round-Trip Reasoning Through Message History

This is the most architecturally delicate step. The engine stores messages as flat `{ role: string; content: string }` objects. `reasoningDetails` is structured data that cannot be serialized into the content string and recovered losslessly (it may contain encrypted blocks, signatures, etc.).

**Two approaches:**

#### Approach A: Extend the Engine's Message Type (Recommended)

Widen the engine's message type to carry optional structured data:

```typescript
// In rlm.ts, change the message type from:
Array<{ role: string; content: string }>

// To:
Array<{ role: string; content: string; meta?: Record<string, unknown> }>
```

The `meta` field carries opaque data that `translateMessages` can use when reconstructing API messages. The engine never inspects it.

In `rlm.ts`, when building the `__TOOL_CALL__` assistant message:

```typescript
messages.push({
    role: "assistant",
    content: `__TOOL_CALL__\n${toolUseId}\n${reasoning}\n__CODE__\n${code}`,
    meta: reasoningDetails ? { reasoningDetails } : undefined,
});
```

In `translateMessages`, when reconstructing the assistant message:

```typescript
if (msg.role === "assistant" && msg.content.startsWith("__TOOL_CALL__\n")) {
    // ... existing parsing ...
    const chatMsg: ChatMessage = {
        role: "assistant",
        content: reasoning || null,
        tool_calls: [{ ... }],
    };
    // Attach reasoning for round-trip
    if (msg.meta?.reasoningDetails) {
        (chatMsg as any).reasoning_details = msg.meta.reasoningDetails;
    }
    result.push(chatMsg);
}
```

**Pros**: Clean separation. Engine doesn't know about reasoning_details. No serialization/deserialization of structured data.

**Cons**: Changes the `CallLLM` signature indirectly (messages type widens). Existing mock `CallLLM` implementations still work because `meta` is optional.

#### Approach B: Serialize reasoning_details into the marker string

Encode `reasoningDetails` as JSON within the `__TOOL_CALL__` marker:

```
__TOOL_CALL__\n{toolUseId}\n{reasoning}\n__REASONING_DETAILS__\n{json}\n__CODE__\n{code}
```

**Pros**: No type changes needed.

**Cons**: Fragile. JSON may contain newlines. Needs escaping. Encrypted reasoning blocks may be large. Ugly.

**Recommendation: Approach A.** The `meta` bag is a minimal, future-proof extension.

## Gotchas and Provider-Specific Concerns

### 1. Not All Providers Return `reasoning`

Only OpenRouter (and providers that support reasoning tokens) will populate `message.reasoning`. Direct OpenAI, Groq, Together, etc. will not. The fallback chain (`reasoning ?? content ?? ""`) handles this gracefully.

### 2. Reasoning Token Costs

Reasoning tokens are billed as output tokens. With `effort: "high"`, the model may generate up to 80% of `max_tokens` in reasoning alone before producing the tool call. For a 16384 max_tokens budget, that's ~13,000 reasoning tokens per call.

**Mitigation**: Make `reasoningEffort` configurable. Use `"medium"` for cost-sensitive runs. Use `"none"` to disable entirely.

### 3. Encrypted Reasoning Blocks

Some providers return `reasoning.encrypted` blocks instead of plain text. These cannot be read but must be round-tripped exactly. The `reasoningDetails` opaque storage handles this -- we never inspect the array, just store and forward it.

### 4. The `parallel_tool_calls: false` Parameter

The local doc (section 4) notes that Chat Completions API does NOT formally include `parallel_tool_calls` in its schema. We send it anyway (line 170). With reasoning enabled, this is even more important to prevent multiple tool calls per response. If OpenRouter ever stops honoring it, we'd need to switch to the Anthropic Messages API or Responses API (which formally support it). For now, the current approach works empirically.

### 5. Message Type Widening

Widening `messages` from `Array<{ role: string; content: string }>` to `Array<{ role: string; content: string; meta?: Record<string, unknown> }>` changes the `CallLLM` type signature indirectly (the `messages` parameter type). However, since `meta` is optional, all existing `CallLLM` implementations (including test mocks) remain compatible without changes.

### 6. Streaming (Future)

The current driver does NOT use streaming. If we add streaming support later, reasoning appears in `delta.reasoning_details` chunks alongside `delta.tool_calls` chunks. The approach described here (opaque storage + round-trip) will work for streaming too -- just accumulate the reasoning_details array from deltas.

### 7. `content` May Become Non-Null

With reasoning enabled, some providers may start returning both `content` (prose text before the tool call) AND `reasoning` (thinking tokens). Our precedence chain (`reasoning ?? content ?? ""`) handles this by preferring `reasoning`. If we want both, we could concatenate: `[reasoning, content].filter(Boolean).join("\n\n")`. Start with reasoning-only and observe behavior.

## Incremental Rollout Plan

1. **Phase 1**: Add `reasoning` to request, parse `reasoning` from response. No round-tripping. This alone fixes the empty-reasoning problem.
2. **Phase 2**: Widen message type with `meta`, round-trip `reasoning_details`. This preserves reasoning continuity across turns.
3. **Phase 3**: Make `reasoningEffort` configurable via CLI/eval harness flags.
4. **Phase 4**: Add driver-level tests.

Phase 1 can ship immediately with minimal risk. Phase 2 is the architectural change that benefits multi-turn agents. Phases 3-4 are polish.
